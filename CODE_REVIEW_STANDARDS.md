## 1. 总则

### 1.1 目的

建立统一的代码审查标准，提升代码质量、可维护性和安全性，降低 defect 率，促进团队知识共享。

### 1.2 适用范围

- 所有提交到 `main` / `master` / `develop` 分支的代码
- 所有 PR（Pull Request）必须经过代码审查
- 紧急修复（hotfix）可走简化流程，但事后需要补审

### 1.3 核心价值观

| 原则 | 说明 |
|------|------|
| **质量优先** | 审查不是为了刁难，是为了守住质量底线 |
| **尊重与建设性** | 指出问题时说明"为什么"，而非"你错了" |
| **自动化先行** | 能用工具检查的，绝不靠人肉审查 |
| **持续改进** | 审查中发现的新问题类型，持续补充到检查清单中 |
| **适度原则** | 避免过度工程化，审查标准与项目阶段和变更规模匹配 |

## 2. 审查角色与职责

### 2.1 角色定义

| 角色 | 职责 | 谁可以担任 |
|------|------|-----------|
| **作者（Author）** | 提交代码，修复审查发现的问题 | 所有开发者 |
| **审查者（Reviewer）** | 逐行审查代码，提出改进意见，最终批准/拒绝 | 资深开发者以上 |
| **维护者（Maintainer）** | 最终合并权限，把控整体方向 | 项目核心成员 |
| **观察者（Observer）** | 旁听学习，不参与正式审批 | 初级开发者 |

### 2.2 责任要求

**作者责任：**
- 提交前自我审查（self-review），确保代码能在本地正常运行
- 提交的代码应遵循项目规范，已通过 ESLint 检查和测试
- PR 描述清晰，说明变更内容、原因和测试方式
- 响应审查意见及时（24 小时内回应）

**审查者责任：**
- 在 **24 小时** 内完成首次审查
- 聚焦代码逻辑正确性、安全性、性能和可维护性
- 不纠结代码格式化（留给自动化工具处理）
- 每个问题说明严重级别（Blocker / Major / Minor / Nitpick）
- 保持友善、建设性的沟通语气

## 3. 审查流程

### 3.1 标准流程

```
[创建 PR] → [CI 自动检查] → [分配审查者] → [代码审查]
    ↓                                                      ↓
[通过] ← [修复确认] ← [作者修复] ← [不通过/需修改]
    ↓
[合并]
```

### 3.2 详细步骤

**步骤 1：作者自检（Pre-commit）**
- [ ] 运行 `npm run lint` 无报错
- [ ] 运行 `npm run build` 或本地启动验证通过
- [ ] 检查是否有调试代码（`console.log`, `debugger` 等残留）
- [ ] Commit message 符合规范

**步骤 2：创建 PR**
- PR 标题格式：`[类型] 简要描述`（参考 [Commit 规范](#11-git-commit-规范)）
- PR 描述模板：

```markdown
## 变更内容
（描述改了什么、为什么改）

## 变更类型
- [ ] 新功能
- [ ] Bug 修复
- [ ] 重构
- [ ] 性能优化
- [ ] 样式更新
- [ ] 文档更新
- [ ] 其他

## 测试方式
（说明如何验证变更的正确性）

## 相关 Issue
Closes #（如有）
```

**步骤 3：CI 自动检查**
- ESLint 检查通过
- 构建通过（`npm run build`）
- 安全扫描（如果有）

**步骤 4：分配审查者**
- 由项目维护者或作者自行选择至少 **1 名** 审查者
- 核心模块（认证、权限、支付）需至少 **2 名** 审查者

**步骤 5：代码审查**
- 审查者自检：
  - [ ] 已在本地运行 `npm run lint` 确认项目基线通过
  - [ ] 检查评论是否与自动工具规则冲突（如 ESLint、Prettier）
  - [ ] 确认每条评论均依据客观标准，而非个人偏好
- 审查者根据检查清单逐项审查
- 标记问题并注明严重级别
- 评论应具体到代码行

**步骤 6：修复与确认**
- 作者根据反馈修复
- Blocking 问题必须全部解决
- 修复后需要审查者重新确认

**步骤 7：合并**
- 所有 Blocking 和 Major 问题已解决
- 至少获得 1 个批准（Approval）
- 由维护者或 PR 作者执行合并
- 建议使用 **Squash Merge** 保持提交历史整洁

### 3.3 紧急流程（Hotfix）

用于生产环境紧急 Bug 修复：

- 可直接合并到 `main` 分支
- 但需至少 1 人即时审查
- 事后 24 小时内补 PR 记录
- 合并后需同步回 `develop` 分支

## 4. 审查级别与评判标准

### 4.1 问题严重级别

| 级别 | 标签 | 含义 | 是否必须修复 |
|------|------|------|-------------|
| **P0 Blocker** | 🔴 | 代码有严重 Bug、安全漏洞、或违反核心架构设计 | **必须修复**，否则拒绝合并 |
| **P1 Major** | 🟠 | 可能导致潜在 Bug、明显代码异味、性能问题 | **必须修复**，否则拒绝合并 |
| **P2 Minor** | 🟡 | 违反规范、可读性差、可优化但不影响功能 | 建议修复，可在后续 PR 中处理 |
| **P3 Nitpick** | ⚪ | 个人偏好、风格建议、小细节优化 | 可选，作者自行决定。应节制使用，每个 PR 不超过 3 条，避免强加个人偏好 |

### 4.2 审查通过条件

| 场景 | 最低要求 |
|------|---------|
| 小型变更（< 50 行） | 无 P0/P1，至少 1 个 Approval |
| 中型变更（50-300 行） | 无 P0/P1，至少 1 个 Approval |
| 大型变更（> 300 行） | 无 P0/P1，至少 2 个 Approval |
| 核心模块变更 | 无 P0/P1，至少 2 个 Approval |
| Hotfix | 无 P0，至少 1 个 Approval |

### 4.3 严重级别判定指南

**Blocker 典型场景：**
- SQL 注入、XSS、CSRF 等安全漏洞
- 硬编码的秘密/密钥/密码
- 用户认证绕过、权限校验缺失
- 死锁、数据库连接不释放
- 导致数据丢失或损坏的逻辑错误
- 直接在生产环境硬编码调试代码

**Major 典型场景：**
- try/catch 中吞掉异常没有日志
- 内存泄漏风险（事件监听未清理、定时器未清除）
- 缺少必要的输入验证
- 大事务长时间持有连接
- N+1 查询问题
- 组件 Props 类型定义不完整
- 重复代码可提取但未提取
- 缺少必要的注释（复杂逻辑无解释）

**Minor 典型场景：**
- 变量命名不够语义化
- 文件命名不符合约定
- 魔法数字未定义为常量
- 缺少或过多空行
- 注释拼写错误或过时
- 未使用的 import 语句

**Nitpick 典型场景：**
- 个人风格的代码组织偏好
- 可选的语法糖替换建议
- 非关键性的变量名建议

## 5. 通用代码规范检查清单

### 5.1 命名规范

| 项目 | 规范要求 | 示例 |
|------|---------|------|
| 后端变量/函数 | camelCase | `userId`, `createTopic()` |
| 后端文件 | camelCase | `authController.js` |
| 后端目录 | 小写复数 | `controllers/`, `routes/` |
| 后端常量/枚举 | UPPER_SNAKE_CASE | `USER_LEVEL.ADMIN`, `HTTP_STATUS.OK` |
| 前端组件文件 | PascalCase | `DebateHeader.vue` |
| 前端非组件文件 | camelCase | `helpers.js`, `useAuth.js` |
| Vue 模板引用 | camelCase | `v-model="loginAccount"` |
| CSS 类名 | kebab-case | `.card-footer`, `.search-box` |
| API 路径 | kebab-case | `/api/debate/topics` |
| 数据库表名 | snake_case（小写） | `debate_topic`, `user_check` |
| 数据库字段 | snake_case（小写） | `user_id`, `audit_status` |

### 5.2 代码组织

- [ ] 单文件职责单一（Single Responsibility）
- [ ] 函数/方法不超过 **50 行**（超过应考虑拆分子函数）
- [ ] 组件模板不超过 **200 行**（超过应考虑拆分子组件）
- [ ] 每个文件不超过 **400 行**
- [ ] 条件嵌套不超过 **3 层**（超过应提前 return 或提取函数）
- [ ] 避免超过 **3 个** 参数（使用对象参数替代）

### 5.3 注释规范

- [ ] 复杂逻辑必须写注释说明"为什么这样做"
- [ ] 注释不是废话（`// 设置变量 i = 1` 是坏注释）
- [ ] TODO 注释必须标注作者和日期：`// TODO(author): 2026-05-03 需要处理边界情况`
- [ ] 公共导出函数应有 JSDoc（或其他文档注释）
- [ ] 不要保留被注释掉的代码——Git 历史已经保留了它们

### 5.4 依赖管理

- [ ] 不使用已被废弃的 npm 包
- [ ] 新引入的依赖需评估：大小、维护状态、许可证兼容性
- [ ] `package.json` 中版本号使用 `^` 或 `~` 精确到小版本
- [ ] 不允许在代码中使用 `require()` 动态加载

## 6. 安全检查清单

### 6.1 认证与授权

- [ ] 所有非公共 API 都经过了 `auth` 中间件校验
- [ ] 权限校验使用了 `requireLevel()` 或 `requireAdmin()`，而非手动比较
- [ ] JWT Token 在响应拦截器中正确处理 401
- [ ] Token 不过期时间合理（建议不超过 7 天）
- [ ] 密码使用 `bcryptjs` 加密存储，不允许明文
- [ ] 敏感操作（如修改密码、绑定邮箱）需要二次验证

### 6.2 输入验证

- [ ] 所有用户输入（body / query / params）都经过了 `express-validator` 校验
- [ ] SQL 查询使用参数化查询（`?` 占位符），禁止字符串拼接
- [ ] 前端提交的数据在后端再次验证（前端验证不能替代后端验证）
- [ ] 文件上传限制大小和类型
- [ ] 对 HTML/JS 内容进行转义（防 XSS）

### 6.3 数据保护

- [ ] 接口不返回敏感字段（密码、Token、内部 ID）
- [ ] 错误信息不暴露内部实现细节（数据库结构、服务器路径等）
- [ ] `.env` / `.env.production` 不在 Git 中（已在 `.gitignore` 中）
- [ ] 日志中不记录密码、Token 等敏感信息

### 6.4 前端安全

- [ ] 所有 API 请求自动携带 Token
- [ ] 前端不在 URL/Storage 中存储敏感信息
- [ ] 使用 `v-html` 时需要特别注意 XSS 风险——非受信内容禁用
- [ ] 路由守卫正确拦截未授权访问

## 7. 后端审查检查清单

### 7.1 路由层（routes/）

- [ ] 路由路径使用 kebab-case
- [ ] 权限中间件链组合正确、顺序合理（auth → validation → permission → audit）
- [ ] URL 参数 ID 使用 `:param` 语法
- [ ] RESTful 设计符合规范（GET 查询 / POST 创建 / PUT 更新 / DELETE 删除）

### 7.2 控制器层（controllers/）

- [ ] try/catch 结构完整
- [ ] catch 块有错误日志（`console.error` 或日志库记录）
- [ ] 使用 `utils/response.js` 统一响应格式
- [ ] 事务处理：`beginTransaction()` → 操作 → `commit()` → catch → `rollback()` → `finally { release() }`
- [ ] 事务中 `release()` 在 `finally` 块中执行，确保一定释放
- [ ] 分页查询使用了 `LIMIT` 和 `OFFSET`
- [ ] 大量数据操作考虑分批处理

### 7.3 中间件层（middlewares/）

- [ ] 中间件签名正确：(req, res, next) 或 (err, req, res, next)
- [ ] 错误处理中间件使用了四个参数
- [ ] 中间件不阻塞正常流程（适当调用 next()）
- [ ] 全局错误处理能覆盖所有异常类型

### 7.4 工具函数（utils/）

- [ ] 纯函数优先，避免副作用
- [ ] 导出的函数有明确输入输出
- [ ] 不直接修改传入的对象（除非明确设计如此）

### 7.5 配置文件（config/）

- [ ] 敏感配置使用环境变量，不硬编码
- [ ] 数据库连接池配置合理（max connections 根据部署环境调整）
- [ ] CORS 配置在生产环境限制来源域名

## 8. 前端审查检查清单

### 8.1 组件审查

- [ ] 组件结构清晰：`<template>` → `<script setup>` → `<style scoped>`
- [ ] Props 定义了类型和默认值：

```javascript
const props = defineProps({
  topicId: { type: Number, required: true },
  items: { type: Array, default: () => [] }
})
```

- [ ] Emits 明确定义事件名：

```javascript
const emit = defineEmits(['update', 'close'])
```

- [ ] 组件作用域样式使用 `scoped`（避免污染全局）
- [ ] HTML 非 void 元素（div、span、textarea等）使用自闭合语法（遵循 `eslint-plugin-vue` 的 `vue/html-self-closing` 规则）
- [ ] 计算属性（`computed`）优于方法（`methods`，或者直接函数调用）用于派生状态
- [ ] `watch` 使用合理，避免深度监听大量数据
- [ ] `onMounted` / `onUnmounted` 成对使用（事件监听、定时器等记得清理）
- [ ] 避免在模板中使用复杂表达式——提取为计算属性或方法

### 8.2 状态管理（Pinia）

- [ ] Store 命名准确，职责单一
- [ ] actions 中的异步操作有错误处理
- [ ] 页面级状态优先使用组件本地状态，全局共享才用 Pinia

### 8.3 API 封装

- [ ] API 调用通过 `api/xxx.js` 封装，不在组件中直接调用 Axios
- [ ] 请求拦截器和响应拦截器配置正确
- [ ] 错误处理统一（如 `handleResponse` 或 Element Plus 消息提示）

### 8.4 性能与用户体验

- [ ] 列表渲染使用了 `key` 属性
- [ ] 大数据列表使用了虚拟滚动或懒加载分页
- [ ] 图片懒加载
- [ ] 避免不必要的大列表重新渲染
- [ ] 异步请求有 loading 状态

## 9. 数据库审查检查清单

### 9.1 SQL 语句

- [ ] 参数化查询（使用 `?` 占位符），禁止字符串插值拼 SQL
- [ ] 查询条件使用了索引字段
- [ ] 大表查询有明确的分页限制
- [ ] `SELECT *` → 显式指定需要的字段
- [ ] JOIN 查询确认了索引，避免全表扫描
- [ ] 批量操作使用 `INSERT ... VALUES (...), (...)`（而非逐条插入）

### 9.2 事务管理

- [ ] 事务范围最小化（不要在不必要的操作中持有事务）
- [ ] 事务中的查询顺序一致（避免死锁）
- [ ] 事务超时时间合理（避免长时间锁行）

### 9.3 数据库设计

- [ ] 字段类型选择合理（不滥用 VARCHAR(255)，整数字段用 INT/BIGINT）
- [ ] 外键约束（如使用）的 ON DELETE 行为明确
- [ ] 索引添加经过分析（非过度索引）

---

## 10. 性能审查检查清单

- [ ] 避免在循环中执行 SQL 查询（N+1 问题）
- [ ] 前端避免不必要的数据派发（watch 深度监听控制）
- [ ] 大文件不要一次性加载到内存
- [ ] 删除不必要的 `console.log`（尤其是生产环境）
- [ ] 静态资源使用懒加载
- [ ] 避免内存泄漏（定时器、事件监听、WebSocket 正确关闭）
- [ ] API 响应数据量合理（不要返回不必要的字段）

## 11. Git Commit 规范

### 11.1 Commit Message 格式

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### 11.2 Type 类型

| Type | 含义 | 是否触发版本发布 |
|------|------|----------------|
| `feat` | 新功能 | 是（minor） |
| `fix` | Bug 修复 | 是（patch） |
| `perf` | 性能优化 | 是（patch） |
| `refactor` | 重构（不修 Bug 不加功能） | 否 |
| `style` | 代码格式（空格、格式化等） | 否 |
| `docs` | 文档更新 | 否 |
| `test` | 测试相关 | 否 |
| `chore` | 构建/工具/依赖相关 | 否 |
| `ci` | CI 配置变更 | 否 |

### 11.3 示例

```
feat(auth): 添加两步验证功能

- 新增 TOTP 生成与验证
- 用户设置页增加 2FA 开关
- 登录流程增加双因素校验步骤

Closes #42
```

```
fix(debate): 修复投票权重计算溢出问题

当用户经验值超过 INT_MAX 时，投票权重计算出现溢出。
改用 BigInt 类型进行计算。

Closes #58
```

```
refactor(api): 统一错误响应格式

将散落在各 controller 中的 res.json({...}) 统一
替换为 utils/response.js 中的 success/error 函数。

无功能变更，仅重构。
```

## 12. 自动化工具配置

### 12.1 必需工具

| 工具 | 用途 | 推荐配置 |
|------|------|---------|
| ESLint | JavaScript 代码静态检查 | 前端：`eslint.config.js`，后端：`eslint.config.js` |
| Prettier | 代码格式化 | `.prettierrc`（全项目统一） |
| EditorConfig | 编辑器基础设置 | `.editorconfig`（间距、编码等） |
| husky | Git Hooks 管理 | `pre-commit` 钩子 |
| lint-staged | 仅检查暂存区文件 | 与 husky 配合 |

### 12.2 可选工具

| 工具 | 用途 | 说明 |
|------|------|------|
| commitlint | Commit message 格式校验 | 配合 `commit-msg` 钩子 |
| markdownlint | Markdown 文件检查 | DOCS 模块可用 |

## 14. 附录：常见问题示例

### 14.1 应避免的坏代码

```javascript
// ❌ 坏：SQL 字符串拼接（SQL 注入风险）
const sql = `SELECT * FROM user WHERE id = ${req.params.id}`

// ✅ 好：参数化查询
const sql = 'SELECT * FROM user WHERE id = ?'
const [rows] = await pool.query(sql, [req.params.id])
```

```javascript
// ❌ 坏：干吞异常
try {
  await someOperation()
} catch (err) {
  // 什么都没做
}

// ✅ 好：日志记录 + 回退处理
try {
  await someOperation()
} catch (err) {
  console.error('操作失败:', err.message)
  throw err
}
```

```javascript
// ❌ 坏：魔法数字
if (user.exp > 1000) { ... }

// ✅ 好：命名常量
const ADVANCED_THRESHOLD = 1000
if (user.exp > ADVANCED_THRESHOLD) { ... }
```

```javascript
// ❌ 坏：深度嵌套
if (a) {
  if (b) {
    if (c) {
      doSomething()
    }
  }
}

// ✅ 好：提前返回
if (!a || !b || !c) return
doSomething()
```

### 14.2 审查者常用评语模板

| 场景 | 评语建议 |
|------|---------|
| 建议修改 | **`[Major]`** 这里存在安全风险，建议使用参数化查询替换字符串拼接 |
| 确认原因 | **`[Minor]`** 我发现这里加了 sleep(1000)，能否解释一下为什么需要它？ |
| 学习建议 | **`[Nitpick]`** 建议了解一下 ES6 解构赋值，这里可以用 `const { name, age } = req.body` |
| 赞赏 | **`[Nice]`** 错误处理做得很到位，特别是考虑了事务回滚的边界情况！ |
