## 2. 关键回顾

### 2.1 需要表扬的 ✅

**M1 — 错误处理不完整 → 准确发现，未被自动化工具覆盖**
- `catch` 块只有 `console.error(err)`，用户无反馈
- 属于"功能正确性"类问题，无法被任何 lint/静态分析工具检测到
- 这类问题是**代码审查对自动化检查的补集价值**的最佳体现

**M2 — 定时器覆盖问题 → 准确发现**
- 组件卸载后重新挂载时 `timer` 被新引用覆盖，旧定时器泄漏
- 在审查标准中已有关于 `onMounted/onUnmounted` 成对的检查项，审查者参考了该清单

**代码整体质量认可**
- Props 类型定义完善、Emits 明确、scoped 样式、无复杂模板表达式
- 这些是审查清单中"✅ 通过"的部分，同样需要记录，不能只记录问题

### 2.2 需要改进的 ❌

**m3 — `textarea` 自闭合判定错误**
- 审查者认为 `textarea` 不应自闭合（基于 HTML 规范认知）
- 但实际 ESLint 配置 `vue/html-self-closing: 'always'` 要求所有元素自闭合
- 教训：**审查者应始终运行实际 ESLint 检查后再给出格式化相关评语**

**n7 — 清理逻辑合并建议脱离实际**
- 建议"合并到一个 cleanup 函数"在当前组件上下文中过度工程化
- 教训：**Nitpick 级别的建议应更有节制，避免强加个人偏好**

### 2.3 审查标准本身发现的问题

#### 问题 A：`vue/html-self-closing` 规则
标准文档中 `8.1 组件审查` 没有提及 HTML 元素自闭合规则。
建议在检查清单中增加：
```
- HTML 非 void 元素（如 div, span, textarea）使用自闭合语法（遵循 ESLint vue/html-self-closing 配置）
```

#### 问题 B：审查流程中缺少"审查者自检"步骤
当前标准流程假设审查者直接"逐行审查"，但未要求审查者：
1. 先在本地运行 ESLint 确认自动检查通过
2. 再确认自己的评论不与自动工具冲突
3. 最终复核所有评论的有效性

建议在"步骤 5：代码审查"中增加审查者自检子步骤。

#### 问题 C：Nitpick 级别的使用边界不够清晰
当前标准中 Nitpick 定义为"个人偏好、风格建议、小细节优化"。
但"审查者应当尊重作者的代码风格决策"未明确写出。
建议补充：
```
Nitpick 建议应有节制。如果建议涉及"个人偏好"多于"客观改进"，
考虑是否真的值得提出。尽量将 Nitpick 限制在每个 PR 不超过 3 条。
```

## 3. 标准文档更新

### 3.1 CODE_REVIEW_STANDARDS.md 更新

#### 版本升级: v1.0 → v1.1

**更新 1：审查流程 — 步骤 5 增加审查者自检（第 3.2 节）**
在"步骤 5：代码审查"中增加：
```
审查者自检：
- [ ] 已在本地运行 npm run lint 确认项目基线通过
- [ ] 检查评论是否与自动工具规则冲突（如 ESLint、Prettier）
- [ ] 确认每条评论均依据客观标准，而非个人偏好
```

**更新 2：前端审查清单 — 追加 HTML 自闭合规则（第 8.1 节）**
在"组件审查"中增加：
```
- [ ] HTML 非 void 元素使用自闭合语法（遵循 ESLint vue/html-self-closing 配置）
```

**更新 3：Nitpick 规则补充（第 4.1 节）**
在 Nitpick 说明中增加：
```
- Nitpick 建议应有节制，每个 PR 不超过 3 条
- 避免强加个人偏好，仅在提供客观改进时使用 Nitpick
```
