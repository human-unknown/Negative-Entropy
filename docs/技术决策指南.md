## 一、全栈核心原则

这些原则高于具体技术决策。当你在两个方案之间犹豫时，回到这里。

### 原则 1：防御性编码是默认姿势

**假设所有输入都是恶意的，直到被证明安全。**

- 后端：每一行 SQL 用参数化查询，每一条用户输入经过 `express-validator`
- 前端：每个 API 返回处理异常路径，每种用户操作考虑边界情况
- 这条原则不做妥协。逆熵的项目审计已经在 `debateController` 中发现过 30+ 条数据库连接重复释放的路径——这就是 **假设代码会出错** 的代价。设计时多怀疑一点，上线后少补一个 Bug。

### 原则 2：显式优于隐式

**读代码的人（包括未来的自己）不应该需要猜测。**

- 函数名表明副作用：不叫 `processUser`，叫 `updateUserLevel`
- SQL 字段显式写出：不写 `SELECT *`，写 `SELECT id, name, level, exp`
- 路由路径明确：`/api/debate/topics` 比 `/api/db/t` 好一百倍
- Emits 明确定义事件名，Props 写清楚类型和默认值

### 原则 3：一致性比完美更重要

**项目里用一种方式，比用"最好的"方式更重要。**

- 后端已用 `utils/response.js` 的 `success()` / `error()` 统一格式，不要某个 controller 自己 `res.json({ code: 200, ... })`
- 前端已用 `@/api/xxx.js` 封装 API，不要在组件里直接 `axios.get()`
- 常量定义在 `constants/` 中，不要到处 `if (status === 1)` —— 魔法数字已经吃过亏了

当你想引入一种新方式时，问自己：**整个项目都这样改，值得吗？** 如果答案是"不"，就别做。

### 原则 4：越靠近用户的操作越需要防御

**逆熵是一个辩论平台。用户互不信任，系统必须绝对可信。**

这条原则决定了权重：
- **用户输入** > **内部数据**：用户发的言论需要 AI 审核 + 人工复核，管理员操作反而可以快一点
- **写操作** > **读操作**：创建辩论比浏览列表需要更严格的权限校验
- **直接影响他人的操作** > **只影响自己的操作**：投票（影响他人胜负）比修改个人头像需要更严格的验证

### 原则 5：每行代码都是 liability

**你不写那行代码，就不需要维护它、测试它、担心它炸。**

- 避免过早抽象：一个函数只在两个地方用到，别急着提取成 util
- 避免过度配置：不要引入你没用到的依赖——这个项目已经做得不错了，`package.json` 很干净
- 避免防御性过度：不是每个 if 都需要 else，不是每个 catch 都需要 console.error（虽然多数情况需要）

## 二、后端：架构与模式

### 2.1 代码组织：三层架构，职责清晰

逆熵后端是经典的三层架构：**路由 → 控制器 → 工具函数**。这不是最好的架构，但它在项目里已经统一了。

#### 路由层（routes/）只做三件事

```javascript
// ✅ 好：路由只做编排
router.get('/topics', auth, validator, controller.getTopics)
router.post('/topics', auth, requireAdmin, validator.createTopic, controller.createTopic)
```

路由层的职责：
1. 挂载中间件链（auth → validator → permission）
2. 将请求映射到 controller 方法
3. **不**包含任何业务逻辑

注意中间件顺序：`auth`（你是谁） → `validator`（输入是否合法） → `permission`（你有没有权限做）。这个顺序不要颠倒了。

```javascript
// ❌ 坏：路由里写了业务逻辑
router.get('/stats', async (req, res) => {
  const [rows] = await pool.query('SELECT ...')
  res.json(success(rows))
})
```

#### 控制器层（controllers/）做什么

控制器是"编排者"，不是"执行者"：

```javascript
// ✅ 好：try/catch 完整，错误有日志，响应用统一格式
export const getTopics = async (req, res) => {
  try {
    const { page = 1, pageSize = 10 } = req.query
    const offset = (page - 1) * parseInt(pageSize)
    const [rows] = await pool.query('SELECT ... LIMIT ? OFFSET ?', [parseInt(pageSize), offset])
    const [countResult] = await pool.query('SELECT COUNT(*) as total FROM ...')
    res.json(success({ list: rows, total: countResult[0].total, page: parseInt(page), pageSize: parseInt(pageSize) }))
  } catch (err) {
    console.error('获取话题列表失败:', err)
    res.json(error('获取话题列表失败'))
  }
}
```

控制器的标准模板：

```javascript
export const actionName = async (req, res) => {
  // 1. 解构输入
  const { param1, param2 } = req.body   // 或 req.query, req.params

  // 2. 前置校验（express-validator 已做结构性校验，这里做业务语义校验）
  if (!param1) return res.json(error('param1 不能为空', 400))

  // 3. 执行业务逻辑（需要事务时用 pool.getConnection()）
  try {
    // ...
    res.json(success(result))
  } catch (err) {
    console.error('操作失败:', err)
    res.json(error('操作失败'))
  }
}
```

**关键：** 控制器里不要直接做响应格式转换、不要管路由拼接、不要重新校验输入（那是 validator 的事）。

#### 事务的黄金规则

```javascript
// ✅ 好：事务模板
const conn = await pool.getConnection()
try {
  await conn.beginTransaction()
  
  // ... 业务操作
  
  await conn.commit()
  res.json(success(result))
} catch (err) {
  await conn.rollback()
  console.error('操作失败:', err)
  res.json(error('操作失败'))
} finally {
  conn.release()  // ⚠️ 一定放在 finally，无论成功失败都释放
}
```

这条规则的教训来自本项目的真实经历：`debateController` 和 `ruleDebateController` 中曾存在超过 30 条路径的事务连接未释放（在多个 return/if 分支中遗漏了 finally 块）。**每一条 try 都必须是 try→catch→finally，没有例外。**

#### 工具函数（utils/）什么时候建

**当一个逻辑在 2 个以上 controller 里出现时，才提取成 utils。**

```
utils/
├── response.js       # ✅ 通用：success/error 所有 controller 都在用
├── auditLog.js       # ✅ 通用：AI审核、人工审核都需记录日志
├── levelUpgrade.js   # ⚠️ 边界：目前只在投票结算后用，但未来多场景需要
├── watermark.js      # ⚠️ 可选：单一用途，是否值得抽象可以讨论
```

工具函数必须是纯函数或有明确文档说明副作用。不要写一个看似纯函数实际上改了全局状态的工具。

### 2.2 数据库操作模式

#### 读操作：保持简单

```javascript
// ✅ 好：参数化查询 + 投影字段
const [rows] = await pool.query(
  'SELECT dt.*, u.name as publisher_name FROM debate_topic dt LEFT JOIN user u ON dt.publisher_id = u.id WHERE dt.audit_status = 1 ORDER BY dt.created_at DESC LIMIT ? OFFSET ?',
  [pageSize, offset]
)
```

**不要做的事：**
- ❌ 字符串拼接 SQL（SQL 注入风险，本项目已全面禁止）
- ❌ `SELECT *` 返回大量不需要的字段（加网络传输和内存开销）
- ❌ 在循环里发 SQL 查询（N+1 问题——前端常见，后端同样存在）

#### 写操作：事务是默认选择

**原则：任何涉及两张表以上的写操作，默认用事务。** 即使你现在觉得不需要原子性：

```javascript
// 创建辩论 -> 需要更新用户统计 -> 需要写入审计日志
// 这三步必须一起成功或一起失败
const conn = await pool.getConnection()
try {
  await conn.beginTransaction()
  const [topicResult] = await conn.query('INSERT INTO debate_topic ...')
  const [statsResult] = await conn.query('UPDATE user_stats ...')
  await conn.query('INSERT INTO audit_log ...')
  await conn.commit()
  res.json(success({ id: topicResult.insertId }))
} catch (err) {
  await conn.rollback()
  console.error('创建辩论失败:', err)
  res.json(error('创建失败'))
} finally {
  conn.release()
}
```

#### 复用连接还是池连接

| 场景 | 使用 | 原因 |
|------|------|------|
| 简单读查询（单表 SELECT） | `pool.query()` | 自动从池获取 + 释放 |
| 复杂关联查询（JOIN + 子查询） | `pool.query()` | 单次查询，无需事务 |
| 需要原子性的写操作 | `pool.getConnection()` + 事务 | 需控制提交/回滚 |
| 多个写操作组合 | `pool.getConnection()` + 事务 | 同一个连接上执行 |

#### 分页的陷阱

```javascript
// ✅ 好：page/pageSize + offset 计算
const page = parseInt(req.query.page) || 1
const pageSize = parseInt(req.query.pageSize) || 10
const offset = (page - 1) * pageSize

// 注意：pageSize 要设上限，防止有人传 pageSize=100000
const MAX_PAGE_SIZE = 100
const safePageSize = Math.min(pageSize, MAX_PAGE_SIZE)
```

**本项目的 `debateController.getTopics` 做得不错**——用 `limit ? offset ?` 参数化分页。但有个小进化空间：给 `pageSize` 加个硬上限，防止滥用。

### 2.3 错误处理策略

#### 三层处理模式

| 层级 | 怎么做 | 目的 |
|------|--------|------|
| 业务逻辑（try/catch） | `console.error` + 用户友好的错误信息 | 不让用户看到 "Cannot read property of undefined" |
| 全局错误处理 | `errorHandler.js` 中间件 | 捕获漏网之鱼 |
| 未捕获异常 | `process.on('uncaughtException')` | 最后防线，记录日志后优雅退出 |

```javascript
// ✅ 好：业务层错误
try {
  await someOperation()
} catch (err) {
  console.error('someOperation 失败:', err.message, err.stack)
  res.json(error('操作执行异常，请稍后重试'))
}
```

**关键规则：每个 catch 必须有 `console.error`。** 项目历史上曾发现 userController 中多处 catch 块缺少错误日志——导致线上问题查不到根因。日志不是为了好看，是为了有人能修。

#### 错误消息：对内 vs 对外

```javascript
// ❌ 坏：把内部实现细节暴露给用户
res.json(error('duplicate entry for key 'idx_user_account''))

// ✅ 好：用户看到友好的，错误记录在内
console.error('注册失败：账号已存在', err)
res.json(error('该账号已被注册'))
```

### 2.4 认证与权限模式

#### 中间件链的约定

路由注册时，中间件顺序固定为：

```
auth → validator(action) → requireLevel/requireAdmin → controller
```

- `auth`：验证 JWT，将 decoded payload 挂到 `req.user`
- `validator(action)`：用 `express-validator` 校验输入格式
- `requireLevel` / `requireAdmin`：检查用户等级权限

#### 权限校验不要手动比较

```javascript
// ✅ 好：使用 requireLevel 中间件
router.delete('/users/:id', auth, requireLevel(USER_LEVEL.ADMIN), adminController.deleteUser)

// ❌ 坏：在 controller 里手动比较
if (req.user.level < 4) {
  return res.json(error('权限不足'))
}
```

**为什么？** 中间件可以集中管理、单元测试、全局修改。手动比较散落在各个 controller 中，改一次权限模型要翻十几个文件。

#### 常量的力量

本项目已经有不错的常量体系：

```javascript
// constants/userLevel.js
export const USER_LEVEL = {
  BEGINNER: 1,
  INTERMEDIATE: 2,
  ADVANCED: 3,
  ADMIN: 4
}

// constants/debateStatus.js
export const DEBATE_STATUS = {
  DRAFT: 0,
  ACTIVE: 1,
  CLOSED: 2,
  ARCHIVED: 3
}
```

**凡是代码中出现 `if (level === 4)` 或 `if (status === 2)` 的地方，都是需要修复的。** 魔法数字要么导致 Bug（拼写错），要么导致混乱（"这个 2 是什么意思？"）。

## 三、前端：组件与数据流

### 3.1 组件设计：一个组件只做一类事

#### 组件分类

逆熵的组件可以分为三类，它们的关注点不同：

| 类别 | 职责 | 示例 | 允许做的事 |
|------|------|------|-----------|
| **展示型** | 接收 props，渲染 UI | `DebateHeader.vue`, `UserLevelCard.vue` | 纯展示，无 API 调用 |
| **交互型** | 处理用户操作 | `SpeechInput.vue`, `VotePanel.vue` | 本地状态 + 调用外部事件 |
| **容器型** | 组合子组件 | `DebateFlow.vue` | 管理数据流 + API 调用 |

**原则：不要在一个组件中混合过多职责。**

```javascript
// ✅ 好：DebateHeader 只做展示，Props 完整
defineProps({
  topic: { type: Object, required: true },
  proCount: { type: Number, default: 0 },
  conCount: { type: Number, default: 0 },
  audienceCount: { type: Number, default: 0 }
})

// ❌ 坏：在展示组件内直接调用 API
onMounted(async () => {
  const res = await fetch(`/api/debate/stats?id=${props.id}`)
  // ... 这不是展示组件该做的事
})
```

#### Props 与 Emits 的规范

```javascript
// ✅ 好：Props 有类型、有默认值、required 明确
const props = defineProps({
  topicId: { type: Number, required: true },
  title: { type: String, default: '未命名话题' },
  tags: { type: Array, default: () => [] }   // ⚠️ 对象/数组默认值用函数返回
})

// ✅ 好：Emits 明确定义
const emit = defineEmits(['vote', 'join', 'report'])

// ⚠️ 需要遵守的：Emits 命名用 kebab-case 在模板中
// <VotePanel @vote="handleVote" />
```

### 3.2 状态管理：何时用 Pinia，何时用 ref

很多团队的问题是 "所有状态都放 Pinia" 或者 "全用 ref 传"。这里有一个清楚的决策树：

1. **状态是否被多个无关组件共享？**
   - 是 → Pinia
   - 否 → 继续看下一条

2. **状态是否应该跨页面保持？**
   - 是 → Pinia（比如用户登录状态）
   - 否 → 组件本地 ref

3. **状态是否从 API 获取且多个页面都需要？**
   - 是 → Pinia + action 获取
   - 否 → 组件本地 onMounted 获取

```javascript
// ✅ Pinia：跨页面共享的状态（用户、主题列表）
// stores/user.js
export const useUserStore = defineStore('user', () => {
  const token = ref(null)
  const user = ref(null)
  const isLoggedIn = computed(() => !!token.value)
  
  async function login(account, password) {
    const res = await authApi.login({ account, password })
    token.value = res.data.token
    user.value = res.data.user
  }
  
  return { token, user, isLoggedIn, login }
})

// ✅ ref：只在当前组件使用的一次性数据
// DebateDetail.vue
const speeches = ref([])
const loading = ref(false)
onMounted(async () => {
  loading.value = true
  speeches.value = await getSpeeches(props.topicId)
  loading.value = false
})
```

**逆熵已做得很好的地方：** `useDebateStatus.js` 是个 composable，把辩论状态相关的计算属性（isActive, canSpeak, canVote, statusText）全部封装起来，各处复用，而且没有任何 Pinia 依赖——纯粹是输入 topic 输出 computed。这种模式就是 Vue 3 推崇的。

### 3.3 API 交互模式

#### 统一通过 api/ 层调用

```javascript
// ✅ 好：所有 API 调用走封装
// api/debate.js
export const getTopics = (params) => request.get('/debate/topics', { params })
export const getTopicDetail = (id) => request.get(`/debate/topics/${id}`)
export const createTopic = (data) => request.post('/debate/topics', data)

// 组件中
import { getTopics } from '@/api/debate'
const topics = await getTopics({ page: 1 })
```

**为什么要这层封装？**
1. URL 路径变更只需改一个文件
2. 可以统一加缓存、去重、重试逻辑
3. 测试时可以 mock 这一层，而不需要 mock Axios

#### 响应处理模式

前端当前用 `handleResponse` 模式：

```javascript
// ✅ 好：统一返回 [data, error] 元组，调用方处理
const [data, error] = await handleResponse(createTopic(form.value), '话题创建成功')
if (data) {
  // 成功处理
} else {
  // error 已在 handleResponse 外可自由处理
}
```

这个模式的好处：**调用方不会忘记处理错误**，因为解除构赋值时 `error` 变量就在那。

#### 401 处理策略

当前 `request.js` 拦截器处理 401 的方式：清除 token → 跳转回首页。

```javascript
if (status === 401) {
  localStorage.removeItem('token')
  localStorage.removeItem('user')
  window.location.href = '/'
}
```

这是一个实用的 **暴力但有效** 策略。未来如果做 SPA 体验优化，可以考虑：
1. 用 refresh token 静默续期
2. 用 router.push 替代 location.href 避免页面硬加载

### 3.4 Composables 模式

Composables 是 Vue 3 最有价值的模式之一。逆熵已有两个好的范例：

**`useDebateStatus.js`** — 纯计算，无副作用
```javascript
export const useDebateStatus = (topic) => {
  const isActive = computed(() => topic.value?.status === DEBATE_STATUS.ACTIVE)
  const canSpeak = computed(() => isActive.value)
  // ...
}
```

**`useContentProtection.js`** — 带生命周期管理
```javascript
export const useContentProtection = () => {
  // ✅ onMounted 注册事件，onUnmounted 销毁
  onMounted(() => enableProtection())
  onUnmounted(() => disableProtection())
}
```

**注意 composable 的生命周期管理原则：**
- `onMounted` 中注册的监听器、定时器，必须在 `onUnmounted` 中清理
- composable 内的 `watch` 如果监听了外部响应式数据，默认会在组件卸载时自动停止（Vue 3 行为）——但 `watchEffect` 不会，需要显式调用 `stop()`

## 四、安全：你的默认思维模型

### 4.1 七大默认不信任

| # | 不信任什么 | 对应措施 |
|---|-----------|---------|
| 1 | 不信任用户输入 | `express-validator` 做输入校验 |
| 2 | 不信任前端传的数据 | 后端独立校验，不要依赖前端说 "这个用户是 admin" |
| 3 | 不信任 SQL 字符串 | 永远用参数化查询 |
| 4 | 不信任 localStorage | 不存敏感信息（密码、Token 等）——Token 是唯一的例外 |
| 5 | 不信任环境变量 | `.env` 文件不在 Git 中，生产环境密钥通过部署工具注入 |
| 6 | 不信任 JS 中的"安全" | 前端的防复制、防截图只防君子不防小人 |
| 7 | 不信任"应该不会" | 出现概率 > 0 的边界情况都要处理 |

### 4.2 前端安全与内容防护的局限

`useContentProtection.js` 做了大量工作：禁止复制、右键、打印、截图、F12。这些是 **温和的劝阻**，不是安全措施。

**关于这个 composable 的决策建议：**
- 如果用户的辩论内容是高度敏感（法律、商业机密），需要后端水印 + 屏幕水印 + 法务条款，前端禁止连心理安慰都算不上
- 如果只是防止随手复制，当前方案够用
- 但 `alert()` 弹窗体验很差——建议改用 `ElMessage.warning()`，毕竟项目已经全站统一了 Element Plus 消息

### 4.3 后端安全的黄金组合

逆熵后端已有的安全措施（很好，保持）：

```
JWT 认证 + bcrypt 密码加密 + 参数化 SQL + express-validator + 速率限制 + 操作审计日志
```

**唯一缺的：** 生产环境的 HTTPS。这不是技术债，是安全债。

## 五、测试：从"写完再测"到"测完再写"

### 5.1 决策原则

| 场景 | 测什么 | 工具 |
|------|--------|------|
| 后端工具函数 | 单元测试——纯函数逻辑 | `node:test` |
| 后端 Controller | 集成测试——模拟请求/响应 | `node:test` + 手动 mock |
| 前端工具函数 | 单元测试 | Vitest |
| 前端组件 | 组件渲染 + 交互测试 | Vitest + `@vue/test-utils` |
| API 接口 | 端到端测试 | 手动 + 脚本 |

### 5.2 写测试的时机

**"写完再测" 只适用于你还在理解问题域的早期探索阶段。**

一旦代码的接口稳定了，规则很简单：

- **新功能** → 先写测试骨架，再实现（即使不遵循严格的 TDD，也应该在 merge 之前测试覆盖关键路径）
- **修 Bug** → 先写一个复现该 Bug 的测试，再修
- **重构** → 先确保已有测试覆盖，再重构，重构完测试全部通过

### 5.3 逆熵的测试现状（截至 v1.0）

当前已有 35 个测试用例，覆盖了：
- 辩论流程端到端 Mock 测试
- 用户注册 API 测试
- 规则管理测试

缺口（来自项目审计）：
- 前端组件的交互测试覆盖率较低
- 后端事务操作的边界测试（rollback 路径）
- 权限中间件的独立单元测试

## 六、代码审查：怎样才算一次好的审查

### 6.1 逆熵的审查体系已有

项目已有完整的 `CODE_REVIEW_STANDARDS.md`（v1.1），包括：
- 问题严重级别体系（P0 Blocker / P1 Major / P2 Minor / P3 Nitpick）
- 七类检查清单（通用 + 安全 + 前后端 + 数据库 + 性能）
- 审查流程（PR → CI → 审查 → 修复 → 合并）
- 通过标准

### 6.2 从"走流程"到"真正有用"的进化建议

**给作者的五个原则：**

1. **PR 不写超过 300 行。** 超过就拆。300 行是人眼有效审查的上限。
2. **PR 描述写清楚"为什么"**。代码本身已经说了"改了什么"，PR 描述要说"为什么这么改"和"不这么改会怎样"。
3. **自我审查再提交。** 运行 `npm run lint` + 本地跑一遍功能，是最基本的尊重。
4. **标签归类明确。** 用 `[feat]` / `[fix]` / `[refactor]` 标记，让审查者知道关注点。
5. **24 小时内响应审查意见。** 长期挂起的 PR 是技术债务。

**给审查者的五个原则：**

1. **先区分"一定要改"和"最好改一下"。** P0/P1 必须改，P2 建议改，P3 尊重作者。
2. **每条评论说"为什么"而不是"你错了"。** "这里没有考虑并发"比"这写法不对"有用十倍。
3. **不纠结格式问题。** ESLint 和 Prettier 不是审查者该操心的。
4. **看到好的代码也说出来。** 只说问题容易让新人变得畏手畏脚。
5. **不超过 3 条 Nitpick。** 多了就变成个人偏好轰炸了。

## 七、决策流程图

### 7.1 后端：这个操作要不要用事务？

```
这个写操作涉及几张表？
  ├── 1 张 → 这个操作本质上是读还是写？
  │          ├── 读 → pool.query()
  │          └── 写 → 操作是否需要回滚？
  │                  ├── 是 → 事务
  │                  └── 否 → pool.query() + 关注依赖一致性
  └── 2 张以上 → 必须用事务
```

### 7.2 前端：这个状态该放哪？

```
这个状态被多个组件共享吗？
  ├── 否 → 它在单个组件内部流转吗？
  │        ├── 是 → ref / reactive
  │        └── 否 → 父→子 props 传递，子→父 emit 事件
  └── 是 → 它需要跨页面保持吗？
           ├── 是 → Pinia
           └── 否 → provide/inject 或 Pinia（看团队习惯）
```

### 7.3 这个功能要不要加测试？

```
这个代码是新的还是改旧的？
  ├── 新的 → 这是内部工具函数还是组件逻辑？
  │         ├── 工具函数 → 必须写单元测试
  │         └── 组件逻辑 → 写组件渲染测试
  └── 旧的 → 修 Bug 还是重构？
             ├── 修 Bug → 先写复现测试再修
             └── 重构 → 确保已有测试覆盖了你要改的部分
```

## 八、附录：项目内模式速查表

### 8.1 已统一的标准

| 项目 | 标准 | 文件位置 |
|------|------|---------|
| API 响应格式 | `{ code, message, data }` | `backend/src/utils/response.js` |
| HTTP 状态码 | `constants/statusCode.js` | `backend/src/constants/` |
| 用户等级枚举 | `USER_LEVEL.*` | `frontend + backend: constants/userLevel.js` |
| 辩论状态枚举 | `DEBATE_STATUS.*` | `frontend + backend: constants/debateStatus.js` |
| 输入校验 | `express-validator` | `backend/src/middlewares/validator.js` |
| API 封装层 | `api/xxx.js` → `request` | `frontend/src/api/` |
| Mock 服务 | `api/mock.js` — 开发模式开关 | `frontend/src/api/mock.js` |
| UI 反馈 | Element Plus `ElMessage` | 全站统一（2026-05-04 迁移完成） |

### 8.2 值得参考的文件模式

| 模式 | 参考文件 | 学习点 |
|------|---------|--------|
| 干净的三层路由 | `authRoutes.js` | 中间件链组合清晰 |
| 好的 Controller | `authController.js` | 标准的 try/catch + 统一响应 |
| 好的 composable | `useDebateStatus.js` | 纯计算、无副作用、可复用 |
| 好的展示组件 | `DebateHeader.vue` | Props 完整、无 API 调用、scoped 样式 |
| 好的常量管理 | `constants/index.js` | 从各文件导出再汇总 |

### 8.3 需要避免的模式

| 反模式 | 问题 | 替代方案 |
|--------|------|---------|
| 事务中缺少 finally release | 连接泄漏 | try→catch→finally 固定模板 |
| controller 中 `if (level === 4)` | 魔法数字 | `USER_LEVEL.ADMIN` |
| 路由中写 business logic | 职责混乱 | 路由只做编排 |
| 组件中直接 `axios.get()` | 难以维护和 mock | 统一通过 `api/xxx.js` |
| 日志中记录密码/token | 安全违规 | 脱敏后再记录 |
